# GitRadar 中文网页前端实施蓝图

## 目标

GitRadar 当前已经具备稳定的日报引擎、规则配置、归档分析和企业微信发送能力，但主要交互仍停留在：

- 手改 `.env`
- 手改 `config/digest-rules.json`
- 手动执行 CLI 命令
- 直接查看归档 JSON

这套方式对维护者是可行的，但对“频繁改规则、校验配置、查看归档和手动补发”的日常工作来说，门槛仍然偏高。

因此网页前端的目标不是做一个新的营销官网，而是做一个本地优先、中文优先的 GitRadar 控制台，让现有能力更容易使用。

## 产品定位

第一阶段前端应定位为：

- GitRadar 的本地中文控制台
- 当前 CLI 和配置体系的可视化外壳
- 单机或小团队内网使用工具
- 面向配置修改、手动执行、归档查看和后续轻反馈扩展

当前阶段不应定位为：

- 独立 SaaS 平台
- 面向公网多租户的后台系统
- 替代现有日报内核的全新实现

## 当前仓库可直接复用的基础

当前仓库已经具备前端控制台最需要的 4 类可复用能力。

### 1. 环境配置边界

环境变量加载和格式校验已经集中在：

- `src/config/env.ts`

这意味着网页前端不需要重新定义一套配置语义，只需要提供更安全的编辑与测试入口。

### 2. 规则配置边界

digest 规则文件和解析校验逻辑已经集中在：

- `config/digest-rules.json`
- `src/config/digest-rules.ts`
- `src/commands/validate-digest-rules.ts`

这意味着网页前端不应自己再写一套“平行校验器”，而应直接复用现有解析与校验逻辑。

### 3. 命令执行入口

当前已有明确的 CLI 入口：

- `npm run validate:digest-rules`
- `npm run generate:digest`
- `npm run analyze:digest -- --date YYYY-MM-DD`
- `npm run migrate:archives`
- `npm run send:wecom:sample`

因此网页前端完全可以先从“执行现有命令并展示日志”起步，而不是重写主流程。

### 4. 归档与复盘资产

当前仓库已经具备本地归档、迁移和分析能力，且架构上明确把归档视为长期资产。网页前端可以直接把归档变成“可浏览、可比较、可筛选”的界面，而不是重新定义存储层。

## 设计原则

- 先做控制台，不先做官网
- 先复用现有内核，不先重写主流程
- 先让配置可编辑、可校验、可执行，再做更复杂的探索体验
- 前端负责交互，后端负责文件、命令和敏感信息边界
- 默认本机 `localhost` 运行，避免一开始引入公网权限模型
- 所有可见执行结果都要保留真实日志，不做“按钮按下即视为成功”

## 推荐技术方案

推荐采用“轻前后端分离”的结构：

- 前端：`Vite + React + TypeScript`
- 本地服务：基于 Node.js 的轻量 HTTP API
- 样式：简洁中文控制台风格，不依赖重型后台模板
- 数据源：本地配置文件、归档文件、现有 TypeScript 配置模块、现有 npm scripts

选择这条路线的原因：

- GitRadar 当前是本地任务驱动项目，不是内容站点
- 首期最重要的是配置管理、命令执行和归档查看
- 相比引入更重的 SSR 框架，这种结构更容易贴合当前 CLI 内核

如果后续网页前端扩展为真正的 Radar 产品，再评估是否切到更完整的全栈框架。

## 建议目录结构

建议在仓库内新增如下结构：

```text
web/
  src/
    app/
    components/
    pages/
    services/
    styles/
  index.html
  vite.config.ts

src/web-api/
  server.ts
  routes/
    config.ts
    commands.ts
    archives.ts
    health.ts
  services/
    env-service.ts
    digest-rules-service.ts
    command-runner.ts
    archive-service.ts
  types/
    api.ts
```

原则上：

- `web/` 只负责浏览器界面
- `src/web-api/` 只负责本地 API、文件读写和命令执行
- 现有 `src/config/*`、`src/commands/*` 继续作为核心业务边界存在

## 页面规划

第一版建议只做 4 个页面，避免范围失控。

### 1. 仪表盘

用途：

- 展示今天是否已有归档
- 展示最近一次校验结果
- 展示最近一次生成和发送结果
- 展示近几天的归档数量和最近更新时间

应包含：

- 今日状态卡片
- 最近运行记录
- 快捷操作按钮
- 最近归档列表

快捷操作建议：

- 校验规则
- 生成日报
- 生成并发送
- 发送样例消息

### 2. 规则配置

用途：

- 可视化编辑 `config/digest-rules.json`

应拆成 5 个编辑区块：

- 主题配置
- 黑名单配置
- 选择策略配置
- 阈值配置
- 权重配置

关键交互：

- `themes` 用表格式或卡片式编辑
- bucket 配置用可增删行表格
- 保存前自动执行后端校验
- 校验错误直接定位到字段路径

### 3. 执行中心

用途：

- 可视化执行现有命令并查看真实输出

应支持：

- `validate:digest-rules`
- `generate:digest`
- `generate:digest -- --send`
- `analyze:digest -- --date ...`
- `send:wecom:sample`
- `migrate:archives`

关键交互：

- 命令运行状态
- 终端日志流
- 退出码展示
- 成功、失败、进行中严格区分

### 4. 归档浏览

用途：

- 浏览 `data/history` 下的日报归档

首期能力：

- 按日期列出归档
- 查看某日 shortlist、候选池、最终入选和排除原因
- 展示规则版本和 schema 信息
- 跳转到分析命令结果

## 后续页面预留

第二阶段后可以补：

- 环境配置页
- 主题过滤页
- 周报视图
- 对比视图
- 轻反馈页

## API 设计建议

本地 API 不应追求大而全，先覆盖前端必需动作即可。

### 配置相关

- `GET /api/config/digest-rules`
  - 读取当前规则配置
- `POST /api/config/digest-rules/validate`
  - 校验前端提交的规则配置草稿
- `PUT /api/config/digest-rules`
  - 落盘保存规则配置

### 环境配置相关

- `GET /api/config/env`
  - 仅返回可展示字段和掩码值
- `PUT /api/config/env`
  - 更新允许编辑的环境配置
- `POST /api/config/env/test`
  - 测试 webhook 或 API 连通性

注意：

- 敏感字段不能默认原文回传到前端
- “读取配置”与“读取掩码视图”必须是不同概念

### 命令执行相关

- `POST /api/commands/validate-digest-rules`
- `POST /api/commands/generate-digest`
- `POST /api/commands/analyze-digest`
- `POST /api/commands/migrate-archives`
- `POST /api/commands/send-wecom-sample`
- `GET /api/commands/:id`

命令执行响应至少应包含：

- `id`
- `status`
- `startedAt`
- `finishedAt`
- `exitCode`
- `command`
- `stdout`
- `stderr`

### 归档相关

- `GET /api/archives`
  - 归档列表
- `GET /api/archives/:date`
  - 单日归档详情
- `GET /api/archives/:date/summary`
  - 轻量摘要

## 前端字段映射建议

规则配置页不要直接展示原始 JSON 文本，而应把主要字段映射为可编辑表单。

### `themes`

映射为：

- 主题名
- 关键词列表
- 添加主题
- 删除主题

### `blacklists`

映射为：

- 描述黑名单词
- README 黑名单词
- topic 黑名单词

### `selection`

映射为：

- 每主题 shortlist 上限
- 每主题候选池上限
- 是否强制保留成熟回暖项目

### `thresholds`

映射为：

- 基础阈值数字输入
- `recentPushMomentum` 表格
- `recentCreationNovelty` 表格

### `weights`

映射为：

- 来源信号权重
- 来源重叠权重
- 成熟度权重
- 覆盖度权重

## 后端实现原则

### 1. 复用现有配置解析逻辑

保存配置前，后端应直接调用 `src/config/digest-rules.ts` 内的解析逻辑完成校验，而不是由前端自己保证格式正确。

### 2. 命令执行统一封装

网页前端触发的所有动作都应通过统一的命令执行器封装，避免前端页面各自拼 shell 参数。

### 3. 日志与状态持久化

执行记录至少应在内存中维护，必要时可落到 `data/runtime/web-jobs/`，便于刷新页面后继续查看状态。

### 4. 敏感信息最小暴露

对于 `.env` 中的敏感项：

- 前端默认只看掩码
- 更新时支持“保持原值”
- 测试连接优先于显示明文

## 安全边界

虽然当前目标是本地控制台，但仍应建立最基本边界：

- 默认仅监听 `127.0.0.1`
- 前端不能直接执行任意 shell
- 后端只允许执行白名单命令
- 所有用户输入先转为结构化参数，再映射到命令
- `.env` 和 webhook 不写入浏览器缓存

如果未来开放到局域网或公网，需要单独补：

- 登录鉴权
- CSRF 防护
- 审计日志
- 细粒度权限模型

## 分阶段实施计划

### 第一阶段：最小可用控制台

目标：

- 看得见当前配置
- 改得动规则配置
- 校验配置是否有效
- 一键触发生成日报

交付范围：

- `web/` 基础前端
- `src/web-api/` 基础服务
- 仪表盘页
- 规则配置页
- 执行中心最小版
- `validate` 和 `generate` 命令接入

完成标准：

- 可以在浏览器中修改规则配置并成功保存
- 可以在浏览器中看到校验错误
- 可以在浏览器中触发日报生成并看到终端输出

### 第二阶段：运维能力补全

目标：

- 不离开网页完成大多数日常操作

交付范围：

- 环境配置页
- 样例发送
- 历史补发
- 归档列表与单日详情
- 命令执行历史

完成标准：

- 常用 CLI 操作都有网页入口
- 敏感配置具备掩码与测试能力
- 归档能在网页中查看而不必翻 JSON

### 第三阶段：Radar 产品化体验

目标：

- 从“管理控制台”升级为“可探索的 GitRadar 前端”

交付范围：

- 归档过滤
- 主题视图
- 项目对比
- 证据卡片
- 周度和月度视图
- 轻反馈入口

完成标准：

- 网页前端不再只是配置面板，而开始承载发现和复盘价值

## 实施优先级

如果只做第一批最值得做的内容，优先顺序建议为：

1. 建立本地 API 服务和命令执行器
2. 做规则配置页并打通保存与校验
3. 做最小仪表盘和生成日报入口
4. 做归档浏览基础页
5. 最后再做 `.env` 编辑

`.env` 编辑排在后面，不是因为不重要，而是因为它涉及敏感信息边界，应该在规则配置和命令执行稳定后再接入。

## 验收标准

第一版网页前端完成时，至少应满足以下验收条件：

1. 能在浏览器里读取当前 digest 规则配置
2. 能在浏览器里修改并保存规则配置
3. 非法配置会在保存前被后端拦截，并返回明确字段错误
4. 能从浏览器触发一次 `validate:digest-rules`
5. 能从浏览器触发一次 `generate:digest`
6. 页面能展示命令真实输出、退出码和完成状态
7. 能浏览已有历史归档列表和某日详情

## 真实验证建议

等网页前端开始开发后，验证顺序建议固定为：

1. 终端直接运行原始命令，确认底层链路正常
2. 通过网页触发同一条链路，确认参数和行为一致
3. 对比终端输出与网页日志输出是否一致
4. 如果涉及发送链路，再额外做真实企业微信验证

这条验证顺序可以避免把“前端没接对”误判为“底层能力失效”。

## 建议的下一步任务拆分

如果开始实现，建议按下面顺序开工：

1. 新增 `src/web-api/server.ts` 和最小健康检查接口
2. 新增统一命令执行器
3. 打通 `digest-rules` 的读取、校验、保存 API
4. 新建前端基础框架和路由
5. 先完成规则配置页
6. 再接入仪表盘和执行中心
7. 最后接入归档浏览

这样可以先把“最有价值的配置修改问题”解决掉，再继续扩展。
