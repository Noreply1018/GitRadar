# GitRadar Blueprint v0.1

## 1. 项目定位

GitRadar 是一个给自己用的每日开源项目雷达。

它不是做“更多项目”，而是做“更值得看的少量项目”。核心价值是帮自己每天快速发现 GitHub 上真正有意思、前沿、值得进一步打开的开源项目。

## 2. 目标用户

唯一目标用户是仓库所有者本人。

因此产品决策优先级如下：

1. 手机上打开就能快速消费
2. 维护成本低
3. 结果少但准
4. 后续可以慢慢演进，不要求一开始做重

## 3. v1 产品目标

每天自动从 GitHub 抓取候选项目，筛出最多 5 个值得关注的项目，通过 Telegram 推送发送到手机，并在本地保存一份历史归档。

已确认的节奏和入口：

- 每天早上 `08:00` 固定推送
- 支持手动触发补抓
- 手动触发同时支持终端命令和 Telegram Bot 指令

## 4. v1 核心能力

### 4.1 数据采集

- 只使用 GitHub 作为第一版数据源
- 定时抓取 GitHub 候选项目
- 允许手动触发补抓

### 4.2 筛选与排序

- 不是简单照搬 GitHub 热门列表
- 需要结合热度、前沿性、趣味性和可探索性做筛选
- 最终每天输出不超过 5 个项目

### 4.3 内容生成

每条项目卡片至少要包含：

- 项目名称和 GitHub 链接
- 一句话说明它是干什么的
- 为什么值得看
- 它的新意或差异点
- 是否适合进一步点进去
- 最近热度变化的简短说明

### 4.4 输出与归档

- 主输出：Telegram 推送
- 辅助输出：本地历史归档
- 仓库内会预留明确目录区分代码、脚本、配置和数据

## 5. v1 非目标

- 多用户账户体系
- 社交、评论、点赞
- 推荐算法平台化
- 多数据源聚合平台
- 重型后台管理系统

## 6. 初步流程

1. 定时任务抓取 GitHub 候选项目
2. 或通过终端 / Telegram Bot 手动触发补抓
3. 用规则做第一轮过滤
4. 用模型对候选项目做摘要和优先级判断
5. 选出当天最终项目
6. 生成 Telegram 推送内容
7. 保存本地归档

## 7. 成本预期

### 7.1 开发成本

如果第一版严格控制在 GitHub + Telegram + 本地归档，整体开发复杂度偏低到中等。

### 7.2 运维成本

如果运行方式采用轻量定时任务，运维成本可以保持很低。

### 7.3 API 成本

模型调用成本主要取决于：

- 每天抓多少候选项目
- 是否每个候选都调用模型
- 是否只对入选前的小批量项目调用模型

为了控制成本，第一版更适合：

- 先用规则缩小候选范围
- 再只对少量候选使用模型

## 8. 当前待定项

1. GitHub 候选项目的具体来源和抓取方式
2. “前沿”和“有意思”的规则定义
3. Telegram 消息版式
4. 本地归档的格式和目录结构
5. 最终运行环境
