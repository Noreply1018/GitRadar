# 推送与交付设计

## 目标

GitRadar 的推送层服务于同一个目标：

- 把同一份结构化 digest 交付到用户真正会看的地方
- 不让具体通道反向污染抓取、筛选和成稿逻辑
- 让日报、告警、补发和后续多渠道扩展共用一套数据结构

## 当前结论

当前正式交付通道以企业微信群机器人为主。

原因：

- 已经真实接入并适合当前用户场景
- 官方能力、接入成本和可维护性之间平衡最好
- 足够承载 GitRadar 当前的日报与失败告警消息

当前阶段不再把 `Telegram` 作为默认第一阶段方案，也不再围绕“多通道先铺开”作为主要规划方向。

## 推送层角色

推送层只负责交付，不负责：

- 抓取候选
- 规则筛选
- 模型成稿
- 决定是否入选

推送层输入应始终是已经完成的 `DailyDigest` 或结构化告警数据。

## 当前实现要求

当前发送器需要满足：

- 支持日常 digest 发送
- 支持失败告警发送
- 支持超长消息分页
- 支持历史归档重发
- 保留清晰日志，便于区分生成发送和归档补发

## 统一接口

发送器继续遵守统一接口：

```ts
interface Notifier {
  sendDailyDigest(digest: DailyDigest): Promise<void>;
}
```

如果后续增加更多通知类型，应保持“统一结构化输入 + 通道适配器输出”的模式，而不是把平台细节写回主流程。

## 建议数据流

1. 定时任务或手动触发启动
2. 抓取候选并补充证据
3. 规则筛选和候选池构建
4. 生成最终 digest
5. 写入归档
6. 交给发送器发送
7. 如果发送失败，记录失败原因并触发告警

这个顺序的关键点：

- 归档优先于发送，避免发送失败时丢失当天产物
- 发送器不应修改 digest 本体
- 重发应直接基于归档，而不是重新抓取和重算

## 当前设计重点

当前推送层的演进重点不是“支持更多平台”，而是：

- 保证企业微信消息稳定可读
- 保证分页、重发和失败告警链路可靠
- 为后续 Web UI、导出卡片和更多通道保留统一 digest 结构

## 后续扩展原则

如果未来增加新通道，必须满足这些前提：

- 有明确用户价值，而不是为了看起来功能更多
- 不改变现有 digest 数据结构的核心语义
- 能复用归档和重发机制
- 不引入与当前项目规模不匹配的运维成本

## 当前决策

GitRadar 的交付层当前收敛如下：

- 企业微信群机器人：正式主通道
- 失败告警：继续复用企业微信群机器人
- 其他通道：仅作为后续扩展位，不进入当前主路线

推送层的第一目标是稳定，不是铺量。
