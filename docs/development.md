# 开发规范

## 当前阶段原则

GitRadar 目前还是文档和蓝图阶段，因此仓库治理先围绕以下目标展开：

- 文档可维护
- 工作流可校验
- 变更可追踪
- 后续接入代码时不需要推倒重来

## 目录约定

- `src/`：应用核心逻辑
- `scripts/`：手动执行入口，例如补抓、重发、导出
- `config/`：配置模板和示例文件
- `data/history/`：历史归档的结构定义和保留说明
- `data/runtime/`：运行期临时数据，不入库
- `data/cache/`：抓取缓存，不入库
- `data/exports/`：导出结果，不入库

根目录尽量只保留仓库级文件，不把脚本、运行结果和临时数据直接堆在顶层。

## CI 原则

在功能代码尚未落地前，CI 重点保证：

- Markdown 文档格式一致
- YAML 文件格式正确
- GitHub Actions 工作流语法有效

等功能代码进入仓库后，再把以下能力并入 CI：

- 单元测试
- 集成测试
- 类型检查
- 真实命令行 smoke test

## 推送与发布

- 代码或架构层面的变化，推送到 `main`
- 文档类变化可以先在本地提交，再按需要推送
- 若仓库已配置 GitHub Actions，则推送后应等待 CI 完成

## 计划中的运行方式

产品层面已确认：

- 每天早上 `08:00` 推送一次
- 支持手动触发
- 手动触发入口同时支持：
  - 终端命令
  - Telegram Bot 指令
