# 架构设计与版本路线

## 定位

GitRadar 当前不是一个通用推荐平台，也不是一个只看热度的榜单脚本。

它的正确定位是：

- 一个面向个人和小团队的开源项目发现雷达
- 一个以“为什么今天值得看”为核心的解释型日报引擎
- 一个正在从每日推送工具演进为“可探索发现系统”的产品

下一阶段的目标不是单纯增加抓取量，而是把 GitRadar 升级为：

`可解释发现层 + 轻反馈排名层 + 可探索归档层`

## 当前版本审计

当前版本已经具备这些稳定能力：

- GitHub Trending、最近更新、最近创建三类候选抓取
- 基于活跃度、新鲜度、成熟度、覆盖度的结构化打分
- 主题推断、多样性限制和成熟项目保留
- 基于候选池的受限模型成稿
- 可重发的本地历史归档
- 企业微信群机器人推送
- 归档分析命令和 GitHub Actions 自动化

当前实现最重要的优点：

- 主流程边界清晰，抓取、筛选、成稿、归档、发送职责分离
- LLM 只负责最终中文编辑，不负责发明 repo、证据或外部事实
- 归档默认保留足够多的上下文，具备复盘价值

当前版本的主要约束和问题：

- 数据源仍然偏少，过度依赖 GitHub 主站热度和 star 信号
- 主题识别仍以静态关键词为主，长期会限制专题和比较能力
- Trending 抓取依赖 HTML 结构，存在脆弱性
- LLM 生成失败时缺少重试、模板降级和人工兜底
- 历史归档存在旧 schema，当前分析命令会走 `legacy` 兼容路径

一句话判断：

GitRadar 已经是一个质量不错的日报生产线，但还不是一个用户会反复探索的雷达产品。

## 设计原则

- 少而准优先于多而杂
- 证据优先于文案
- 结构化信号优先于模型自由发挥
- 长期可复盘优先于一次性推送
- 轻量演进优先于过早平台化

## vNext 产品方向

GitRadar 下一阶段应该围绕这三个方向展开：

### 1. 轻反馈个性化

当前 GitRadar 只有公共规则，没有用户反馈闭环。

下一阶段应引入最小反馈集合：

- `saved`：收藏，说明值得持续跟踪
- `skipped`：跳过，说明相关性不足
- `opened`：点开查看，说明标题和证据有效
- `deep-dived`：深挖，说明值得进入个人长期关注列表

这一层的目标不是立刻做复杂推荐系统，而是为后续个性化排序准备真实行为数据。

### 2. 证据可视化

当前“为什么是现在”主要以文字表达为主。

下一阶段应把证据升级为可视化卡片：

- 证据标签：Trending、近期更新、近期创建、成熟项目再升温
- 趋势概览：star 变化、最近更新时间、活跃窗口
- 对比卡片：为什么 A 入选、B 落选
- 主题聚类：同一轮候选的主题分布和去重结果

这会直接提高可信度、复盘效率和传播效果。

### 3. 专题集合与长期档案

GitRadar 不应只输出“今天的 6 到 8 条”。

下一阶段需要增加长期组织层：

- `Collections`：固定主题集合
- `Tracks`：连续几周持续升温的方向
- `Weekly Review`：从日数据提炼周变化
- `Monthly Map`：沉淀出主题迁移和结构性变化

日报负责发现，专题负责积累，归档负责复盘。

## 目标架构

GitRadar 的目标架构建议拆成 6 层。

### 1. Source Layer

负责候选发现与补充证据。

当前：

- GitHub Trending
- GitHub Search recently updated
- GitHub Search recently created

后续扩展：

- release 信号
- issue / PR 活跃度
- contributor 变化
- ecosyste.ms 等外部索引

### 2. Scoring Layer

负责规则筛选、主题识别和候选池构建。

要求：

- 保留结构化打分
- 支持配置化权重
- 支持主题多样性与成熟项目保留
- 支持“新秀”“成熟回暖”“基础设施转折”等不同 lane

### 3. Editorial Layer

负责最终日报成稿。

要求：

- 模型只在受限候选池中工作
- 模型只使用系统提供的证据
- 模型失败时要能退回模板化输出
- 对生成结果做严格校验和最小字段保证

### 4. Archive Layer

负责把每日结果沉淀为长期资产。

要求：

- 归档必须带 schema version
- 历史数据可迁移，不长期依赖 `legacy` 降级路径
- 保存候选、shortlist、候选池、最终入选和排除原因
- 为后续 compare、collections、weekly review 提供基础数据

### 5. Delivery Layer

负责把相同 digest 发送到不同渠道。

当前：

- 企业微信群机器人

后续：

- 增加更多发送器时，仍复用统一 digest 数据结构
- 发送器不反向污染抓取、筛选和成稿逻辑

### 6. Feedback Layer

负责收集用户行为并反哺排序。

初期只需要：

- 本地事件日志
- 基础偏好画像
- 轻量 rerank

这一层是 GitRadar 从“日报工具”进化为“发现系统”的关键。

## 版本推进路线

### V1.2

目标：把现有日报引擎打磨成稳定产品内核。

重点任务：

- 增加归档迁移工具，统一旧 schema
- 为抓取和成稿增加失败重试与降级输出
- 将规则权重、主题定义、阈值改为配置化
- 增加更多硬信号，如 release、issue/PR 活跃度、star 增速
- 补齐真实链路验证记录和运维脚本

交付标准：

- 每日链路具备更强的稳定性
- 分析命令默认面对统一归档结构
- 文档、归档、规则版本三者能对齐

### V1.3

目标：从日报生成器升级为可探索的 Radar。

重点任务：

- 增加 repo compare 能力
- 增加 Radar Card 和证据可视化
- 增加 Collections / Tracks / Weekly Review
- 支持按照主题、时间、成熟度过滤历史归档

交付标准：

- 用户可以探索“为什么是它”
- 用户可以对比“为什么不是另一个”
- 日报不再是唯一出口，归档开始具备产品价值

### V2.0

目标：引入轻个性化和行为反馈。

重点任务：

- 建立 `opened / saved / skipped / deep-dived` 事件模型
- 实现轻量个性化 rerank
- 为后续浏览器扩展或本地 UI 预留反馈入口
- 提供“相似项目”与“推荐给你的原因”

交付标准：

- GitRadar 开始具备个体差异
- 同一批候选在不同用户视角下可重排
- 推荐逻辑不再只有公共规则

### V2.x

目标：从 GitHub 日报进化为开源生态雷达。

重点任务：

- 引入跨平台索引或补充源
- 纳入安全、依赖、赞助、生态关系等信号
- 形成更稳定的专题和趋势观察能力

## 推荐优先级

如果只做最值得做的三件事，优先级如下：

1. 统一归档 schema，并把分析能力从兼容模式拉回主路径
2. 增加失败重试、模板降级和更稳健的抓取链路
3. 做证据可视化和专题集合，而不是继续单纯增加规则数量

## 非目标

当前阶段明确不做：

- 多 agent 编排驱动的复杂采编系统
- 重数据库、重任务队列的平台化改造
- 一开始就追求全自动个性化推荐
- 为了“看起来更大”而盲目接入大量低质量数据源

## 参考项目与启发

这些项目对 GitRadar 的下一阶段设计有直接启发：

- [GitRec](https://github.com/gorse-io/gitrec)：轻反馈、相似项目、行为驱动推荐
- [OSS Insight](https://github.com/pingcap/ossinsight)：把开源发现做成可探索分析产品
- [repos.ecosyste.ms](https://repos.ecosyste.ms/)：多源开源生态索引和元数据沉淀
- [Daily Stars Explorer](https://github.com/emanuelef/daily-stars-explorer)：时间线、比较视图、趋势证据
- [GitHubDaily](https://github.com/GitHubDaily/GitHubDaily)：长期编辑部能力和主题沉淀
- [Star History](https://github.com/star-history/star-history)：轻量趋势图和可分享证据图表

## 当前决策

GitRadar 接下来的默认推进顺序收敛如下：

1. 先把日报链路做稳，完成归档统一和失败降级
2. 再把归档做成可探索资产，而不是只留 JSON
3. 再引入轻反馈，逐步演进为个性化雷达

这份文档是当前版本的主架构设计与版本路线，后续所有产品和架构讨论默认以此为基线更新。
