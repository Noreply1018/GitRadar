# 架构设计与版本路线

## 定位

GitRadar 当前不是一个通用推荐平台，也不是一个只看热度的榜单脚本。

它的正确定位是：

- 一个面向个人和小团队的开源项目发现雷达
- 一个以“为什么今天值得看”为核心的解释型日报引擎
- 一个已经具备轻反馈闭环与轻个性化能力的本地产品

当前阶段的目标不是单纯增加抓取量，而是继续把 GitRadar 打磨成：

`可解释发现层 + 轻反馈排序层 + 可探索归档层 + 本地控制台`

## 当前版本审计

GitRadar 2.0 已经具备这些稳定能力：

- GitHub Trending、最近更新、最近创建三类候选抓取
- 基于活跃度、新鲜度、成熟度、覆盖度的结构化打分
- 主题推断、多样性限制和成熟项目保留
- 基于候选池的受限模型成稿
- 可重发、可迁移、可分析的本地历史归档
- 企业微信群机器人推送
- 本地中文控制台
- `收藏 / 稍后看 / 跳过` 反馈闭环
- 兴趣轨迹与偏好学习提示
- GitHub / LLM / 企业微信环境可用性指纹

一句话判断：

GitRadar 已经不只是一个日报生产线，而是一个开始形成长期使用闭环的开源兴趣雷达。

## 设计原则

- 少而准优先于多而杂
- 证据优先于文案
- 结构化信号优先于模型自由发挥
- 长期可复盘优先于一次性推送
- 轻量演进优先于过早平台化
- 半自动建议优先于黑盒自动推荐

## 当前目标架构

GitRadar 当前建议维持 6 层架构边界。

### 1. Source Layer

负责候选发现与补充证据。

当前：

- GitHub Trending
- GitHub Search recently updated
- GitHub Search recently created

后续扩展：

- release 信号
- issue / PR 活跃度
- contributor 变化
- 更稳的外部生态索引

### 2. Scoring Layer

负责规则筛选、主题识别和候选池构建。

要求：

- 保留结构化打分
- 保持配置化权重
- 支持主题多样性与成熟项目保留
- 支持“兴趣主题”“探索位”“成熟回暖位”等不同 lane

### 3. Editorial Layer

负责最终日报成稿与阅读引导。

要求：

- 模型只在受限候选池中工作
- 模型只使用系统提供的证据
- 模型失败时可退回模板化输出
- 对生成结果做严格校验
- 归档顶部允许规则化总编前言

### 4. Archive Layer

负责把每日结果沉淀为长期资产。

要求：

- 归档必须带 schema version
- 历史数据可迁移，不长期依赖 `legacy` 降级路径
- 保存候选、shortlist、候选池、最终入选和排除原因
- 为后续 compare、tracks、weekly review 提供基础数据

### 5. Delivery Layer

负责把相同 digest 发送到不同渠道。

当前：

- 企业微信群机器人

要求：

- 发送器不反向污染抓取、筛选和成稿逻辑
- 发送成功与失败要保留明确验证记录

### 6. Feedback Layer

负责收集用户行为并反哺排序。

当前已落地：

- 本地事件日志
- 基础偏好画像
- 同仓库与同主题轻量 rerank
- 兴趣轨迹
- 偏好学习提示

下一阶段重点：

- 继续提高提示质量
- 保持克制，不直接演化成黑盒推荐系统

## 版本推进路线

### v2.0.0

目标：把 GitRadar 从“日报生成器 + 控制台”推进成“个人开源兴趣雷达”。

已完成重点：

- 建立 `saved / later / skipped` 事件模型
- 收藏页提供兴趣轨迹
- 归档页提供总编前言与偏好学习提示
- 环境页提供最近成功验证的可用性指纹
- README、展示页和 release 素材已能以真实前端截图展示产品状态

### v2.x

目标：提高长期探索能力，而不是盲目扩面。

重点方向：

- compare：比较两天或两轮候选差异
- tracks：追踪连续升温的方向
- weekly review：从日数据提炼周变化
- collections：围绕固定主题沉淀长期观察
- 更好的证据可视化，而不是只靠长文本解释

### v3.0

目标：如果真的进入下一阶段，不是“再加几个页面”，而是把长期档案、专题沉淀和可解释探索做成真正的产品主界面。

触发条件应包括：

- 日报与反馈数据已形成足够稳定的长期价值
- 用户开始重复查看旧归档而不只是当天日报
- 当前本地控制台信息架构已接近上限
