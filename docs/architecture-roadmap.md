# 架构设计与演进方向

## 产品定位

GitRadar 不是通用推荐平台，也不是只看热度的榜单脚本。

它的定位是：

- 面向个人和小团队的开源项目发现雷达
- 以“为什么今天值得看”为核心的解释型日报引擎
- 具备反馈闭环、归档沉淀和本地控制台的长期使用工具

## 当前稳定能力

仓库当前已经具备这些核心能力：

- GitHub Trending、最近更新、最近创建三类候选抓取
- 基于活跃度、新鲜度、成熟度和覆盖度的结构化打分
- 主题推断、多样性限制和成熟项目保留
- 基于候选池的受限模型成稿
- 可迁移、可分析、可重发的本地归档
- 企业微信群机器人推送
- 本地中文控制台
- `收藏 / 稍后看 / 跳过` 反馈闭环
- 兴趣轨迹与偏好提示
- GitHub / LLM / 企业微信环境可用性指纹

## 设计原则

- 少而准优先于多而杂
- 证据优先于文案
- 结构化信号优先于模型自由发挥
- 长期可复盘优先于一次性推送
- 轻量演进优先于过早平台化
- 半自动建议优先于黑盒自动推荐

## 目标架构

### 1. Source Layer

负责候选发现与补充证据。

当前来源：

- GitHub Trending
- GitHub Search recently updated
- GitHub Search recently created

扩展时优先补强信号质量，不优先扩来源数量。

### 2. Scoring Layer

负责规则筛选、主题识别和候选池构建。

要求：

- 保留结构化打分
- 保持配置化权重
- 支持主题多样性与成熟项目保留
- 保持对兴趣主题和探索位的克制控制

### 3. Editorial Layer

负责最终日报成稿与阅读引导。

要求：

- 模型只在受限候选池中工作
- 模型只使用系统提供的证据
- 模型失败时可退回模板化输出
- 对生成结果做严格校验
- 允许规则化的总编前言，但不引入无约束自由发挥

### 4. Archive Layer

负责把每日结果沉淀为长期资产。

要求：

- 归档必须带 schema version
- 保存候选、shortlist、候选池、最终入选和排除原因
- 为后续 compare、tracks、weekly review 提供数据基础

### 5. Delivery Layer

负责把同一份 digest 交付到目标通道。

当前正式通道：

- 企业微信群机器人

要求：

- 发送器不反向污染抓取、筛选和成稿逻辑
- 发送成功与失败要保留明确验证记录

### 6. Feedback Layer

负责收集用户行为并反哺排序。

当前范围：

- 本地事件日志
- 基础偏好画像
- 同仓库与同主题轻量 rerank
- 兴趣轨迹
- 偏好学习提示

## 接下来优先做什么

当前优先级不是展示包装，而是产品内核：

- 提高推荐准确性
- 提高证据完整度
- 提高控制台与主流程一致性
- 提高归档的长期复盘价值

不优先推进：

- 为了扩面而盲目接更多低质量数据源
- 为了复杂而复杂的多 agent 采编系统
- 过早引入数据库、队列和平台化基础设施
- 面向公开展示的额外包装资产
