# 版本管理说明

## 当前版本阶段

GitRadar 当前正式进入：

`v2.0.0 · 个人开源兴趣雷达`

这个阶段的核心不是单纯把日报生成出来，而是把：

- 日报生成
- 本地控制台
- 归档阅读
- 轻反馈闭环
- 轻个性化提示
- 环境可用性指纹

收口成一套可长期使用的本地产品。

## 分支策略

- 默认主分支为 `main`
- 功能和架构改动默认通过特性分支推进
- 提交到远端时默认先开 PR，再等待受保护分支所需检查通过
- 不再把直接推送 `main` 作为日常开发主流程

## 提交策略

采用简洁的 Conventional Commits 风格：

- `feat:` 新功能
- `fix:` 问题修复
- `docs:` 文档更新
- `chore:` 杂项维护
- `refactor:` 重构
- `test:` 测试相关
- `style:` 纯格式或样式整理

## 版本语义

GitRadar 当前版本路线以产品能力边界而不是单个脚本里程碑划分：

- `1.0.x`：基础日报引擎、发送链路与归档
- `1.1.x`：证据化日报与分析能力
- `1.2.0`：规则配置化、失败降级、归档迁移
- `1.3.x`：本地中文控制台、Docker、Windows 双击启动
- `2.0.0`：轻反馈、轻个性化、编辑型归档、环境指纹
- `2.x`：把“发现”进一步推进为“长期兴趣轨迹”和“专题化雷达”

## 版本升级原则

### Patch

适用于：

- bug 修复
- 推送链路稳定性修复
- 文案和文档修正
- 不改变主能力边界的小幅增强

### Minor

适用于：

- 新增明确能力
- 规则结构升级
- 归档结构增强并提供迁移路径
- 新分析命令、新探索入口或新控制台视图

### Major

适用于：

- 产品定位变化
- 主架构发生明显调整
- 数据模型或交互模式升级到新阶段
- 从“工具链补丁”变成“下一阶段完整产品”

## 发布节奏

阶段性节点应明确打 tag 和 release，尤其是：

- 完成真实可用的用户链路
- 完成一个明确版本目标的收口
- 完成架构层面的阶段升级
- 完成 README / 展示页 / release 页面统一切口

## 文档与版本同步

- 每次阶段性版本变化都应更新 `CHANGELOG.md`
- `README.md` 应直接反映当前最终产品状态，不写发布说明口吻
- 展示页、传播文案、Release 封面和版本号必须同步
- 版本路线和架构决策以 [架构设计与版本路线](./architecture-roadmap.md) 为准
- 如果旧规划文档与当前路线冲突，应直接覆盖旧规划，不保留并行口径

## 发布流程建议

默认主流程：

1. 本地完成改动
2. 运行必要检查和测试
3. 如果涉及用户可见链路，补真实终端验证
4. 如果涉及 README 展示，更新截图资产
5. 推送到特性分支
6. 创建 Pull Request 并等待 CI 完成
7. 合并到 `main`
8. 在确认阶段性完成后打 `vX.Y.Z` tag
9. 创建 GitHub Release
