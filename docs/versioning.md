# 版本管理说明

## 分支策略

- 默认主分支为 `main`
- 在项目早期可以直接在 `main` 上推进
- 当实现代码逐渐增多后，再根据需要切特性分支

## 提交策略

采用简洁的 Conventional Commits 风格：

- `feat:` 新功能
- `fix:` 问题修复
- `docs:` 文档更新
- `chore:` 杂项维护
- `refactor:` 重构
- `test:` 测试相关

## 版本策略

项目早期采用 `0.x.y` 版本号：

- `0.1.0`：第一版明确蓝图和基础仓库治理
- `0.2.0`：完成可运行的最小原型
- `0.3.0`：补齐核心自动化链路
- `1.0.0`：产品和流程稳定后再考虑

## 发布节奏

在蓝图阶段，不强求频繁打 tag。

建议在以下节点打版本：

- 完成一轮明确的产品蓝图
- 完成第一个可运行原型
- 完成第一个稳定自动推送版本

## 变更记录

- 每次有明确阶段性变化时更新 `CHANGELOG.md`
- 文档类演进也应留痕，方便回顾思路变化

## 远端仓库建议

- 建议尽快绑定 GitHub 远端仓库
- 本地完成较完整阶段后推送到 `main`
- 如果后续引入 CI，再将“推送后等待 CI 通过”纳入默认工作流
