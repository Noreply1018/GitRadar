# 版本管理说明

## 分支策略

- 默认主分支为 `main`
- 在项目早期可以直接在 `main` 上推进
- 当实现代码逐渐增多后，再根据需要切特性分支

## 提交策略

采用简洁的 Conventional Commits 风格：

- `feat:` 新功能
- `fix:` 问题修复
- `docs:` 文档更新
- `chore:` 杂项维护
- `refactor:` 重构
- `test:` 测试相关

## 版本策略

当前版本进入 `1.x.y`：

- `0.1.0`：完成第一版蓝图和基础仓库治理
- `1.0.0`：打通本地与 GitHub Actions 自动发送，完成真实企业微信验证，并具备按日期重发已有归档的能力
- 后续 `1.x.y` 版本继续围绕“稳定日用、内容质量、可维护性”演进

## 发布节奏

阶段性节点应明确打 tag 和 release，尤其是：

- 完成真实可用的用户链路
- 完成自动化运行验证
- 完成面向用户的稳定版本收口

## 变更记录

- 每次有明确阶段性变化时更新 `CHANGELOG.md`
- 文档类演进也应留痕，方便回顾思路变化

## 发布流程建议

默认主流程：

1. 本地完成代码、测试和真实终端验证
2. 推送到 `main`
3. 等待 CI 通过
4. 对涉及自动化推送的版本，再补一次 GitHub Actions 手动触发验证
5. 打 `vX.Y.Z` tag
6. 创建 GitHub Release
