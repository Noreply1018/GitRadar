# 版本管理说明

## 分支策略

- 默认主分支为 `main`
- 当前仓库仍可直接在 `main` 上推进
- 当并行工作明显增多时，再按需要切出特性分支

## 提交策略

采用简洁的 Conventional Commits 风格：

- `feat:` 新功能
- `fix:` 问题修复
- `docs:` 文档更新
- `chore:` 杂项维护
- `refactor:` 重构
- `test:` 测试相关

## 版本语义

GitRadar 当前的版本推进不再只围绕“把日报发出去”，而是围绕下面这条路线：

- `1.1.x`：证据化日报的稳定收口
- `1.2.0`：统一归档 schema、增强失败降级、规则配置化
- `1.3.0`：可探索 Radar，增加 compare、专题集合和证据可视化
- `2.0.0`：引入轻反馈与轻个性化
- `2.x`：从 GitHub 日报进化为开源生态雷达

## 版本升级原则

### Patch

适用于：

- bug 修复
- 推送链路稳定性修复
- 文案和文档修正
- 不改变主能力边界的小幅增强

### Minor

适用于：

- 新增明确能力
- 规则结构升级
- 归档结构增强并提供迁移路径
- 新分析命令或新探索能力

### Major

适用于：

- 产品定位变化
- 主架构发生明显调整
- 数据模型或交互模式升级到新阶段

## 发布节奏

阶段性节点应明确打 tag 和 release，尤其是：

- 完成真实可用的用户链路
- 完成一个明确版本目标的收口
- 完成架构层面的阶段升级

## 文档与版本同步

- 每次阶段性版本变化都应更新 `CHANGELOG.md`
- 版本路线和架构决策以 [架构设计与版本路线](./architecture-roadmap.md) 为准
- 如果旧规划文档与当前路线冲突，应直接覆盖旧规划，不保留并行口径

## 发布流程建议

默认主流程：

1. 本地完成改动
2. 运行必要检查和测试
3. 如果涉及用户可见链路，补真实终端验证
4. 推送到 `main`
5. 涉及代码或架构改动时等待 CI 完成
6. 在确认阶段性完成后打 `vX.Y.Z` tag
7. 创建 GitHub Release
